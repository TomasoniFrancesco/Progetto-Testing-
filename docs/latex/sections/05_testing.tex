\subsection{Esecuzione e copertura}

\screenshot{junit_run_all_green}{Esecuzione di tutta la suite JUnit}{%
In Eclipse (o IntelliJ), tasto destro sulla cartella \texttt{src/test/java} $\to$
\textbf{Run As $\to$ JUnit Test}, oppure da terminale \texttt{mvn test}. Screenshot
della vista JUnit con tutti i test verdi e il conteggio totale (es. ``Runs: N/N,
Errors: 0, Failures: 0'').}

La copertura è stata misurata con JaCoCo (\texttt{mvn verify}, report in
\texttt{target/site/jacoco/index.html}). Dal report sono esclusi la UI Vaadin
(testata end-to-end con Selenium, non misurabile da JaCoCo) e la classe di avvio
Spring Boot (3 righe di boilerplate); l'esclusione è dichiarata nel \texttt{pom.xml}.
Alla prima misurazione il package \texttt{core} risultava al 98\% di instruction e
95\% di branch coverage, con \texttt{Parcheggio} già al \textbf{100\%/100\%} —
coerente con la verifica ESC dei contratti (\S\ref{sec:jml-ref}) — e 3 branch
scoperti in \texttt{SmartParkingFSM}: il self-loop di \texttt{gestisciUsc}
(\texttt{transito\_ok} falso in USC), il ramo \texttt{display == null} di
\texttt{cambiaStato}, e il \texttt{default} dello switch di \texttt{step}. I primi
due erano veri buchi di test e sono stati chiusi aggiungendo due test mirati
(\texttt{uscRestaInUscFinoAlTransito}, \texttt{fsmFunzionaSenzaDisplay}). Il terzo è
\emph{strutturalmente irraggiungibile}: lo switch gestisce tutti e 8 i valori
dell'enum \texttt{StatoSistema}, quindi il \texttt{default} (difensivo, lancia
\texttt{IllegalStateException}) non può essere eseguito — è l'analogo Java dei rami
infattibili già osservati con AsmetaV sul modello (\S\ref{sec:asmetav-vc}). Dopo
l'aggiunta dei due test, l'unico branch non coperto del package è proprio quel
\texttt{default}.
